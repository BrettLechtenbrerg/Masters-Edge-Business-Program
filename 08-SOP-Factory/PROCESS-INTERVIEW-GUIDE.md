# Process Interview Guide
## Part of The SOP Factory

---

## Overview

**Purpose:** A comprehensive question bank for extracting process knowledge from subject matter experts (usually the business owner or experienced team member).

**When to Use:**
- Documenting complex processes
- When the main SOP Factory prompt needs more depth
- Interviewing team members about their processes
- Auditing existing processes for improvement

**Time Required:** 30-60 minutes per process

---

## The Interview Mindset

### For the Interviewer (You or AI)

```
┌─────────────────────────────────────────────────────────────────┐
│                    INTERVIEW PRINCIPLES                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. ASSUME NOTHING                                              │
│     What seems obvious to them is NOT obvious to a new person   │
│                                                                 │
│  2. ASK "WHY" OFTEN                                            │
│     Understanding the reason helps document better              │
│                                                                 │
│  3. SEEK SPECIFICS                                             │
│     "Process the order" → "Click the green 'Process' button"   │
│                                                                 │
│  4. FIND THE EXCEPTIONS                                        │
│     The happy path is easy; edge cases are where SOPs fail     │
│                                                                 │
│  5. UNCOVER TRIBAL KNOWLEDGE                                   │
│     "Oh, everyone knows that" = most important to document      │
│                                                                 │
│  6. STAY CURIOUS, NOT JUDGMENTAL                               │
│     "Why do you do it that way?" not "Why would you do that?"  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### For the Subject Matter Expert

Share this with whoever you're interviewing:

> "I'm going to ask you detailed questions about how you do [process]. The goal is to document this so thoroughly that someone brand new could follow it. Please share everything - even things that seem obvious to you. The 'obvious' stuff is usually what new people struggle with most. There are no wrong answers - I just want to capture how this actually works."

## Interview Question Bank

### Section 1: Process Identity

**Basic Information**
```
□ What is this process called? (Official and informal names)
□ In one sentence, what does this process accomplish?
□ Who is the "customer" of this process? (Who benefits from it being done?)
□ How long has this process existed? Has it changed significantly?
□ Who designed or created this process?
□ Is this process documented anywhere currently?
```

**Importance & Impact**
```
□ Why does this process matter to the business?
□ What happens if this process is done poorly?
□ What happens if this process isn't done at all?
□ How does this process connect to other processes?
□ What metrics or KPIs does this process affect?
```

---

### Section 2: Process Boundaries

**Scope Definition**
```
□ What STARTS this process? (Trigger event)
□ What ENDS this process? (Completion criteria)
□ What's included in this process?
□ What's explicitly NOT included? (Boundaries)
□ At what point does this process hand off to another process?
```

**Frequency & Volume**
```
□ How often does this process occur?
   - Daily / Weekly / Monthly / Per customer / On demand / Other
□ How many times per [period] does this happen?
□ Is there a busier time? Slower time? Seasonal patterns?
□ How much variation is there in volume?
```

---

### Section 3: Prerequisites & Setup

**Before Starting**
```
□ What needs to be true before this process can begin?
□ What information must you have before starting?
□ What decisions must already be made?
□ What previous processes must be complete?
□ Are there any timing dependencies? (Must happen before X, after Y)
```

**Tools & Access**
```
□ What systems, software, or tools do you use?
□ For each tool:
   - What is it used for in this process?
   - Where do you access it?
   - What login/credentials are needed?
   - Any tips for using it effectively?
□ What physical items or equipment are needed?
□ What templates or forms do you use?
□ Where are these resources stored/found?
```

**Permissions & Authority**
```
□ Who has permission to do this process?
□ What authority does the person need?
□ Are there approval requirements built in?
□ What decisions can they make independently?
□ What requires escalation?
```

---

### Section 4: The Core Process (Deep Dive)

**Main Flow Discovery**
```
□ Walk me through this process as if you're doing it right now.
□ What's the very first thing you do?
□ Then what? [Repeat until completion]

For EACH step, ask:
□ What exactly do you do? (Specific actions)
□ Why do you do it this way?
□ What tool or system do you use?
□ How long does this step take?
□ What are you looking for/checking?
□ How do you know this step is complete?
□ What could go wrong at this step?
```

**Hidden Steps**
```
□ Is there anything you do automatically that you might forget to mention?
□ Any "quick checks" you do between steps?
□ Do you ever do steps in a different order? When?
□ Are there any steps you sometimes skip? When is that okay?
□ What do you do while waiting for something? (If applicable)
```

**Parallel Activities**
```
□ Are any steps happening simultaneously?
□ Do you ever start something and come back to it later?
□ Are there notifications or follow-ups you set during the process?
```

---

### Section 5: Decision Points

**Identifying Decisions**
```
□ Where in this process do you have to make a judgment call?
□ What are the decision points that aren't black and white?
□ Where does the process branch into different paths?
□ What do you consider when making each decision?
```

**Decision Deep Dive** (For each decision point)
```
□ What is the decision you're making?
□ What factors do you consider?
□ What are the possible options?
□ For each option:
   - When would you choose this?
   - What are the implications?
   - What happens next?
□ What if you're unsure? Who do you ask?
□ Can you give me examples of each path?
□ What's the most common path?
□ What's the rarest path?
```

---

### Section 6: Exceptions & Edge Cases

**Exception Discovery**
```
□ What are the unusual situations you sometimes encounter?
□ Tell me about a time this process didn't go smoothly.
□ What makes some instances harder than others?
□ What variables change how you handle this?
□ Are there VIP customers, special cases, or unique situations?
```

**Exception Handling** (For each exception)
```
□ Describe the exception
□ How often does this happen? (Common / Occasional / Rare)
□ How do you recognize when this is happening?
□ What do you do differently?
□ Is there a completely different process for this?
□ Who else needs to be involved?
□ What's the outcome for the customer/business?
```

**Escalation Paths**
```
□ What situations require escalation?
□ Who do you escalate to?
□ How do you escalate? (Method)
□ What information do you provide when escalating?
□ What's the expected response time?
□ What do you do while waiting for escalation response?
```

---

### Section 7: Quality & Verification

**Quality Standards**
```
□ What does "done well" look like?
□ What does "done poorly" look like?
□ What's the difference between acceptable and excellent?
□ Are there specific standards or criteria?
□ How is quality measured?
```

**Verification Methods**
```
□ How do you check your own work?
□ Is there a review or approval step?
□ Who reviews this? What do they look for?
□ How do you know if something needs to be redone?
□ What would cause a rejection or redo?
```

**Error Detection**
```
□ What mistakes are easy to make?
□ How would you know if something went wrong?
□ What errors have you seen or made?
□ How were they caught?
□ How were they fixed?
```

---

### Section 8: Communication & Handoffs

**Internal Communication**
```
□ Who else is involved in or affected by this process?
□ When do you need to communicate with them?
□ What information do you share?
□ What method do you use? (Email, Slack, in person, etc.)
□ Any templates or standard messages?
```

**External Communication**
```
□ Do you communicate with customers/vendors/external parties?
□ At what points in the process?
□ What do you communicate?
□ Any templates or scripts?
□ What's the expected tone or style?
```

**Handoffs**
```
□ Does this process hand off to another person or team?
□ At what point does the handoff occur?
□ What information do they need from you?
□ How do you transfer the work?
□ How do they know it's their turn?
□ Who's responsible during the transition?
```

---

### Section 9: Documentation & Tracking

**Record Keeping**
```
□ What do you document during this process?
□ Where do you record/log activities?
□ What information must be captured?
□ What format is used?
□ How long are records kept?
□ Who needs access to these records?
```

**Status Tracking**
```
□ How do you track where something is in the process?
□ How do others know the status?
□ Is there a tracking system or dashboard?
□ How do you handle work in progress?
```

---

### Section 10: Timing & Deadlines

**Time Expectations**
```
□ How long does this entire process take?
   - Best case scenario
   - Typical case
   - Worst case scenario
□ What takes the most time?
□ What's surprisingly quick?
□ Where are the bottlenecks?
```

**Deadlines & SLAs**
```
□ Are there deadlines or service level agreements?
□ What are the time-sensitive elements?
□ What happens if a deadline is missed?
□ How do you prioritize when things are backed up?
□ What can be done to speed up the process when needed?
```

---

### Section 11: Improvement & Automation

**Pain Points**
```
□ What's frustrating about this process?
□ What would you change if you could?
□ What takes too long?
□ What feels unnecessary?
□ What do you wish was automated?
```

**Automation Opportunities**
```
□ Which steps are repetitive and predictable?
□ Which steps could software do?
□ What triggers could start automated actions?
□ What currently requires manual copying between systems?
□ What notifications could be automatic?
```

**Past Improvements**
```
□ Has this process been improved before?
□ What changed? What drove the change?
□ Did the changes work? Why or why not?
□ What improvements have been suggested but not implemented?
```

---

### Section 12: Training & Onboarding

**Learning Curve**
```
□ How long does it take to learn this process?
□ What's hardest for new people to understand?
□ What mistakes do new people commonly make?
□ What do you wish you'd known when you started?
```

**Training Approach**
```
□ How were you trained on this?
□ How have you trained others?
□ What works well for training?
□ What training materials exist?
□ What's missing from current training?
```

**Support Needs**
```
□ How much support do new people need?
□ How long until they're independent?
□ What questions do they ask most often?
□ When do they need to ask for help vs. figure it out?
```

## Interview Flow Template

Use this structure for a complete interview:

```
┌─────────────────────────────────────────────────────────────────┐
│                    PROCESS INTERVIEW AGENDA                     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   OPENING (5 min)                                              │
│   □ Introduce purpose of interview                             │
│   □ Explain how documentation will be used                      │
│   □ Set expectations for detail level                          │
│                                                                 │
│   SECTION A: THE OVERVIEW (10 min)                             │
│   □ Process Identity questions                                  │
│   □ Process Boundaries questions                                │
│                                                                 │
│   SECTION B: THE WALK-THROUGH (20-30 min)                      │
│   □ Prerequisites & Setup                                       │
│   □ Core Process (step by step)                                │
│   □ Decision Points                                             │
│                                                                 │
│   SECTION C: THE EDGE CASES (10-15 min)                        │
│   □ Exceptions & Edge Cases                                     │
│   □ Quality & Verification                                      │
│                                                                 │
│   SECTION D: THE ECOSYSTEM (5-10 min)                          │
│   □ Communication & Handoffs                                    │
│   □ Documentation & Tracking                                    │
│                                                                 │
│   SECTION E: THE FUTURE (5 min)                                │
│   □ Pain Points & Improvements                                  │
│   □ Training Considerations                                     │
│                                                                 │
│   CLOSING (5 min)                                              │
│   □ Review key points captured                                  │
│   □ Identify any gaps to follow up                             │
│   □ Discuss next steps                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## Interview Prompt for Claude Code

If you want to use Claude Code to conduct the interview, use this prompt:

```
You are a Process Documentation Specialist conducting an interview to extract everything needed to create a comprehensive SOP.

Your interview style is:
- Curious and thorough - you ask follow-up questions
- Patient - you let them fully explain before moving on
- Specific - you push for concrete details, not generalities
- Non-judgmental - you're there to document, not critique

Interview structure:
1. Start with overview questions to understand the big picture
2. Walk through the process step-by-step
3. Return to probe decision points and exceptions
4. Explore quality checks and communication
5. End with improvement opportunities

Key behaviors:
- When they say "just," "simply," or "obviously" - probe deeper
- When they give a general answer - ask for a specific example
- When they mention a decision - ask about all possible paths
- When they skip ahead - back them up to capture what was missed
- When they're done - summarize and ask what was missed

At the end, generate a structured summary organized for SOP creation.

Begin the interview by asking: "What process are we documenting today? Give me the name and a one-sentence description of what it accomplishes."
```
